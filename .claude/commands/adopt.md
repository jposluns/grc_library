Invoke the `adopt` skill defined in this project's pack at [`dev-security/claude-rules/skills/adopt/SKILL.md`](../../dev-security/claude-rules/skills/adopt/SKILL.md). Slash-command entry point is `/adopt`; the underlying skill name remains `adopt`. This is the RUN-ONCE onboarding for a fork that adopts this governance-corpus-plus-pack project: it resets the machinery-core `.working/` working-state (the audit-trail registers, the session handoff and its queue, the closed-work ledger, the pending-decisions and verifier-override queues, the deep-assessment register, the per-document review anchors) to clean adopter baselines, settles how the adopter will handle the absent sibling repositories, strips maintainer-only operational residue, and records the adopter's choices in a committed `.claude/adopt-config.json` so the resume step proceeds in adopter-mode without re-asking. It runs ONLY on a confirmed adopter clone (a fork origin), never on the maintainer's own repo or a maintainer's fresh-machine clone, and only once (an existing adopt-config short-circuits it). It never touches the corpus, the pack, the tooling, the gates, or the version lineage, which are the product the adopter came for. Execute the onboarding per the seven-step process the skill encodes:

1. **Confirm this is an adopter clone, and that adoption has not already run**: run `tools/detect-env.py` and read its `operator_identity`; proceed ONLY if `adopter`. Stop on `maintainer` (the maintainer's repo) or `maintainer-fresh-machine` (advise cloning the siblings, not resetting working-state), and stop if a `.claude/adopt-config.json` already exists (already adopted). Confirm the classification with the operator before any reset.

2. **Choose the sibling model**: ask the adopter, recording the choice, whether they will keep their **own siblings** (their own reference-base / worker-exchange / private-operational repositories beside the clone) or run **self-contained** (on the in-repo `.ref`/`.scratch`/`.private` placeholders plus the reference-acquisition manifest). Surface both options with consequences; do not pick silently.

3. **Reset the machinery-core working-state to clean adopter baselines**: for each machinery-core `.working/` surface (the session lease + handoff, the validation-sweep / validate-pr / retrospective / credit-offload / reference-audit registers, the closed-work ledger, the next-actions projection, the pending-decisions and verifier-override queues, the overnight-work file, the deep-assessment register, and the detailed CHANGELOG current week) clear the maintainer's accumulated DATA to an empty or stub baseline while preserving the SHAPE each gate validates (headers, metadata, table headers, lifecycle fields). Seed nothing else in the first cut; do not reset the corpus, pack, tooling, gates, root CHANGELOG, or version surfaces.

4. **Handle the sibling placeholders per the chosen model**: for **own siblings**, record the adopter's sibling locations for the resolver and the resume step, and optionally bootstrap the adopter's EXTERNAL `grc_library_ref` sibling from the reference-acquisition manifest (fetch the freely-and-legally-available sources INTO that external repository, list the licensed/blocked for manual acquisition, never redistributing licensed content); for **self-contained**, keep the in-repo placeholders as-is and use the manifest as a bibliography to build an external reference base later. In BOTH models fetched reference text goes into an EXTERNAL sibling, NEVER into the in-repo `.ref` stub: the in-repo placeholders always stay stub-only so the stub-guard gate passes (a fetch into `.ref` would hard-fail it and break step 7's green sweep).

5. **Strip maintainer-only machinery**: remove or neutralize residual maintainer-only operational content (private-operational-repository pointers, maintainer contact and watermark tokens, maintainer-specific runbook references the corpus and gates do not depend on). Do not touch the pack, the gates, or the corpus; this is scoped to operational residue, not product.

6. **Record the adopt-config**: write and commit `.claude/adopt-config.json` with the minimal schema (`mode` = `"adopter"`, `adopted_at` = the UTC date, `sibling_choice` = `"own-siblings"` | `"self-contained"`, `adopt_config_version` = `1`). The committed config is the durable marker that this fork is adopted, read by step 1 and the resume step to skip re-onboarding.

7. **Verify and report**: confirm the reset clone still passes the full audit sweep sibling-free (the adopter-clone portability invariant), confirm the resume step now reports adopter-mode and does not re-propose `/adopt`, and report what was reset, the sibling model recorded, and the committed adopt-config location. If a gate fails after the reset, a surface was cleared past its required shape; restore that shape and re-run before declaring the onboarding complete.

Do not run this on a non-adopter clone (step 1 gates it), do not fabricate adopter history (baselines are EMPTY, not invented), and do not re-run on an already-adopted fork without explicit governance-authority confirmation (the committed adopt-config short-circuits it).

Report back: the confirmed operator classification, the sibling model chosen, the working-state surfaces reset (and any placeholder or manifest handling), the maintainer-only residue stripped, the committed adopt-config, and the green sibling-free sweep confirming the reset clone is adopter-ready.
