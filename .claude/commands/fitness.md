Invoke the `library-fitness-review` skill defined in this project's pack at [`dev-security/claude-rules/skills/library-fitness-review/SKILL.md`](../../dev-security/claude-rules/skills/library-fitness-review/SKILL.md). Slash-command entry point is `/fitness`; the underlying skill name remains `library-fitness-review` (the descriptive name documents the workflow's purpose, the slash command is the ergonomic verb). Execute the corpus-wide ten-persona library-fitness review per the nine-step process the skill encodes:

1. **Establish mechanical baseline**: run `tools/run_all_audits.sh` standalone and capture the result. If any gate fails, fix the underlying defect first, then return to this step. The fitness review starts from a corpus that passes mechanical audit.

2. **Identify scope and run ordinal**: scope is whole corpus (every page reachable from the library's top-level index). The run ordinal is the next `rN` after the most recent entry in [`.working/fitness-reviews/history.md`](../../.working/fitness-reviews/history.md). Multiple runs in one calendar day increment `r1 → r2 → r3`.

3. **Dispatch ten persona subagents in parallel**: each persona receives the persona-specific brief (canonical source: [`.working/fitness-reviews/README.md`](../../.working/fitness-reviews/README.md) § Personas) with scope, focus questions, and explicit exclusions. **Strip maintainer mental model from each brief** — fresh-reader requires fresh context. The ten personas are: (1) first-time executive reader, (2) security practitioner, (3) GRC practitioner, (4) auditor/assurance reviewer, (5) policy and standards editor, (6) process owner/operational user, (7) skeptical reader, (8) adoption practitioner, (9) privacy/data protection officer, (10) newcomer/onboarding engineer. **All ten dispatched on every full run**; skipping a persona is the inference-cascade failure mode that the pack's 7th rule (`validate-inference-before-action.md`) catches at the dispatch decision. The only sanctioned exception is a maintainer-authorised scoped run recorded in the history-row Summary cell. Each persona must return findings as structured blocks (persona identifier, ruleId, severity, location, evidence, impact, recommendation, retention decision); reject any finding lacking quoted evidence or a specific location pointer and re-dispatch.

4. **Synthesise findings** via the six-step synthesis rubric: (4.1) dedupe by `(file_path, normalised_section, claim_type)`; (4.2) tag each synthesised finding `R | I | K` (Real evidence / Inference / Known-issue resurfaced); (4.3) adjudicate severity onto `High [critical] | High | Medium | Low | FYI` (pick higher for adjacent disagreement; invoke debate at step 4.5 for larger divergence); (4.4) record persona provenance per finding; (4.5) debate for divergence >1 severity level OR real-vs-false-positive disagreement; (4.6) the history row's `Personas` column declares dispatch (Rule 5.6-equivalent for fitness reviews).

5. **Verify findings, then triage by severity tier** in four sub-steps before any finding lands in the remediation backlog: (5.1) write the report with all findings marked `verification: unverified`; (5.2) Pass-1 orchestrator verification — re-read each cited source location and apply a verdict tag `✅ confirmed-as-stated` / `⚠️ confirmed-with-modification` (with inline modification) / `❌ rejected` (with inline rejection rationale) / `🤔 ambiguous-needs-maintainer`; (5.3) Pass-2 maintainer-interactive bucket processing — the `✅` cluster gets a batch confirmation, the `⚠️` cluster gets per-finding prompts with the recommended adjustment plus alternatives, the `🤔` cluster gets per-finding prompts with the open question, the `❌` cluster gets a batch presentation with optional per-finding escalation; (5.4) triage and severity-tier action for confirmed findings only — High [critical] → must-fix-before-next-major-reliance-event; High → must-fix-this-quarter; Medium → should-fix-next-review-cycle; Low → routine cleanup; FYI → informational. Confirmed findings produce TODO entries carrying the `FR-<n>` ID, the originating run reference, and the Pass-2 verification date. **Trust-recovery routing flag** (step 5.5): when `/fitness` runs as the trust-recovery suite's second pass, step 5.4's normal triage is replaced by the tier's routing convention (every confirmed finding tiered by severity: High[critical]/High to P1, Medium/Low to P2, tagged `[fitness]`, with no Low/FYI deferral and nothing dropped, maintainer sign-off retained); outside trust-recovery mode the normal triage applies unchanged.

6. **Write the combined report** to [`.working/fitness-reviews/`](../../.working/fitness-reviews/) at `YYYY-MM-DD-rN.md`. Eight top-level H2 sections in this order: `## Executive Summary`, `## Review Method`, `## Page-by-Page Findings`, `## Cross-Library Findings`, `## Severity Model`, `## Recommendations`, `## Standardization Recommendations`, `## Remediation Backlog` (with discrete `FR-<n>` IDs). Optional `## Final Assessment` if sections 1, 6-8 leave material to summarise. **Only when findings exist**; zero-finding runs leave only the history-row trace.

7. **Append the history-row** to [`.working/fitness-reviews/history.md`](../../.working/fitness-reviews/history.md) with columns `Date | Run | Personas | Findings | Resulting PR | Detail | Summary`. New row on top (reverse-chronological). Zero-finding runs still get a row; the `Detail` column is `none` and `Resulting PR` is `none` for those.

8. **Surface issues, recommendations, and choices in chat**: the actionable layer of the fitness review is the prioritisation conversation, not the file. After writing the report, surface: each High [critical] finding (with location and one-line action recommendation), the top 5 High findings, the cross-library patterns surfaced, the standardisation recommendations with effort estimates, and the remediation backlog count for prioritisation.

9. **Maintain follow-up tracking**: once the maintainer triages and assigns remediation IDs to PRs, update the Open remediation backlog table in `history.md` with each item's status and assigned PR. When a PR closes a remediation item, the PR's CHANGELOG entry references the `FR-<n>` ID so the trail back to the originating review is preserved.

Reject any persona finding that lacks an explicit `path:line` quote or a specific location reference. A finding without quoted evidence is a hypothesis, not a finding; re-dispatch the persona with the evidence requirement re-emphasised.

Report back: the audit baseline result, the per-persona findings (grouped by persona, with `path:line` evidence per finding), the synthesised findings (grouped by severity), the triage decisions, the path to the combined report, the new history row, and the prioritised recommendation set for maintainer action.
