Invoke the `high-assurance-verification` skill defined in this project's pack at [`dev-security/claude-rules/skills/high-assurance-verification/SKILL.md`](../../dev-security/claude-rules/skills/high-assurance-verification/SKILL.md). Slash-command entry point is `/high-assurance`; the underlying skill name remains `high-assurance-verification` (the descriptive name documents the workflow's purpose, the slash command is the ergonomic verb). This is the heavier pre-apply verification harness for a SENSITIVE change, one that is gate-blind on correctness, delicate at scale, and costly to get wrong. It is distinct from the content sweeps (`/validate`, `/validate-pr`), the semantic-fit audit (`/matrix-fit`), and the reactive trust-recovery suite (`/full-qa`, `/fitness`): those review the corpus, a window, or control-code fit; this produces pre-apply assurance for one high-stakes change through independent adversarial verification and a deterministic apply. Use it before applying such a change, when the maintainer directs absolute-integrity rechecking, or when resuming a sensitive item left open in the register. Execute the harness per the seven-step process the skill encodes:

1. **Confirm the trigger and open the register row**: confirm all three conditions hold (gate-blind correctness, delicate scale, high escaped-error cost); if only one or two hold, stop and use the routine layers. When uncertain, prefer the harness and record the judgement. Open a row in [`.working/high-assurance/register.md`](../../.working/high-assurance/register.md) (item, which conditions, status `in-progress`) so it survives a session boundary. Apply guard-first sequencing: if a mechanical gate for the value class will exist, land the gate before the data.

2. **Research fan-out**: dispatch parallel research workers to produce the candidate values (research, not final prose; the orchestrator re-reads and authors). Record which workers ran.

3. **Mechanical signal pass over the negatives**: run a deterministic search over the candidate set, targeting the negatives (items marked "no fit" / "N/A" / "no change"). Route every negative carrying a signal of the target attribute to a verifier read (the open-on-negative-with-signal discipline): a negative on a candidate bearing a signal must be grounded in opening the source, not inferred.

4. **Independent adversarial verification**: dispatch separate verifier workers, blind to each other and to the research reasoning, briefed to refute not confirm. Two complementary lenses: a false-negative verifier (re-read the negatives, hunt the misses) and a false-positive verifier (judge each assigned value against the source title, hunt the over-assignments). Each verdict quotes ground truth. A miss loops back: corrected and re-verified before the apply, not patched after.

5. **Programmatic invariant floor**: run every deterministic check the value class admits (identifier in the canonical set, count matches, flag set) and require it to pass. This is the existence-gate floor under the semantic verification; it does not replace steps 4 or 6.

6. **Deterministic scripted apply, then re-parse**: do NOT hand-edit the sensitive artefact. Drive the apply with a dry-run-validated, idempotent-guarded script keyed on an explicit source-to-value map, then re-parse the rendered artefact and confirm every applied value matches the map (zero mismatches). The script running is not evidence of correctness; the re-parse is. The script and its dry-run output are part of the record.

7. **Close the register row and surface**: record the stages and outcomes in the register row (workers, verifier findings, invariant results, apply script and re-parse result) and set status `verified`; surface the findings in chat. Do not silently drop a `pending` / `in-progress` row; if the item cannot complete this session, leave it `pending` / `in-progress` (or `deferred` with the blocker named) so the next `/resume` re-surfaces it.

The harness is NOT a gate and NOT for routine changes: the three-condition trigger is a gate, not a default-on, and indiscriminate use erodes the signal that an item is genuinely sensitive. It does not relax the routine apply-time verification; it adds independent adversarial verification and a deterministic apply on top.

Report back: the trigger confirmation and register row, the research workers dispatched, the negatives signal-passed, the two verifiers' findings (misses and over-assignments, each with quoted ground truth), the invariant-floor result, the scripted apply and its re-parse result (mismatches, if any), and the register row's final status.
