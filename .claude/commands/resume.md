Resume the previous session from the durable handoff record. This is the single command the maintainer sends to continue work in a fresh session.

Execute these steps in order:

1. **Read [`.working/session-handoff.md`](../../.working/session-handoff.md) in full**, including its **"Known environment behaviours"** section. It is the as-of-last-refresh snapshot of branch, versions, counts, last-merged PRs, trust-recovery state, the next-actions queue, open decisions awaiting the maintainer, the known environment behaviours, and the standing disciplines. The known-behaviours section matters: the stop-hook auto-commits and pushes uncommitted changes on turn-end (the working tree is auto-persisted, not held locally), so verify `git log` rather than assuming the tree is uncommitted.

2. **Read [`.claude/CLAUDE.md`](../CLAUDE.md)** (the PRIMORDIAL RULE and project disciplines), the most recent few entries of [`CHANGELOG.md`](../../CHANGELOG.md) to ground recent history, and **[`.working/third-party-issues.md`](../../.working/third-party-issues.md)** (the log of execution-environment / third-party-service issues) so a recurring environment flake (e.g. a commit-signing-server 503 reddening gate 36) is recognized as an artefact, not chased as a corpus regression.

3. **Verify state against live files, do not trust the snapshot blindly** (the snapshot drifts forward between refreshes): run `git rev-parse --is-shallow-repository` (if `true`, `git fetch --unshallow` before any history-aware audit); `git log --oneline -5` to confirm HEAD; `tools/run_all_audits.sh` to confirm the corpus is green; and check the library/pack/README versions in `README.md` and `dev-security/claude-rules/README.md` against the snapshot. **Confirm the mechanical close-vs-start diff**: the `run_all_audits.sh` result on live `main` should match the snapshot's **green-at-`<sha>`** line (same gate count, same sha or a descendant). A mismatch here is the one genuine deterministic close-vs-start signal (e.g. a shallow-clone history-gate artefact, or a merge that dropped something), as distinct from the non-deterministic `/validate` subagent layer; investigate it before proceeding.

4. **Surface a one-screen orientation to the maintainer**: current verified state (branch, HEAD, versions, gate/rule/skill counts), what merged most recently, and the next 1-3 planned actions from the queue. Note any drift between the snapshot and live state.

5. **Batch the unattended-run clarifications up front (maintainer-directed, 2026-06-25).** Before resuming the queue, ask the maintainer, via `AskUserQuestion`, batched into as few rounds as possible, every decision whose answer would let the session run unattended longer, so the assistant does not have to stop and ask mid-run. Pre-populate maintainer decisions for: (a) **all Priority 1 and Priority 2 backlog items** in [`TODO.md`](../../TODO.md) (the disposition, approach, and any ambiguous external value for each, drawing the candidate options from the item's recorded notes and a quick read of the target artefacts so the options are concrete); and (b) **the next 10-plus planned PRs**, including any that fall outside P1 and P2 (scheduled larger tracks such as the FR-167 batches, or P3-and-lower items the running order will reach). Group related items so the maintainer answers in a handful of structured questions rather than dozens; lead each with a recommended option. Record the answers in the next-actions queue (and, for durability across a mid-session compaction, in the working state) so they survive the rest of the session. Where the maintainer defers a decision, mark it deferred and route around it rather than blocking. The cost of one batch of clarifications at session start is far lower than repeatedly pausing an otherwise-unattended run. Apply the **compute-don't-ask** gate first: do not ask what the assistant can verify itself (versions, counts, file state); ask only the genuine authorial decisions.

6. **Run a full corpus-wide `/validate` as the first substantive task**, before resuming the queue. (It is fine to issue the step-5 clarification batch and then run this sweep while the answers come back; the sweep does not depend on the answers.) This is the compensating control for the session-closing handoff PR, which intentionally skips its trailing `/validate-pr` to avoid the post-merge validate-then-PR loop (a handoff PR's `/validate-pr` would produce ledger rows that need a new PR, whose merge would trigger another `/validate-pr`, with no terminating "next substantive PR" at a session boundary). The corpus-wide `/validate` is stronger than the skipped per-PR sweep would have been: it re-examines the whole corpus, not just the handoff PR's diff, so anything the prior session's closing PR(s) might have introduced is caught here. Record its findings per the `/validate` skill and route any to the backlog before continuing. **Cross-check the findings against the handoff's "Asserted expectations" section**: a finding that *contradicts* a surface the closing session asserted clean (a claimed-clean touched surface) is a genuine miss signal (the closing session's self-assessment was wrong), to be escalated rather than triaged as an ordinary out-of-window finding; a finding on a surface the closing session did not assert clean is an ordinary finding under the normal triage. This is why the closing session records asserted expectations rather than running a second `/validate` at close (the corpus is byte-identical across the boundary, so a two-run diff would fire on subagent sampling variance, not corpus facts).

7. **Resolve pending decisions before the queue (attended-autonomous mode).** Read [`.working/pending-decisions.md`](../../.working/pending-decisions.md). If its queue is non-empty, resolve each entry BEFORE starting the next queued items: surface each "proceeded with stricter-safe default" entry to the maintainer for confirm-or-redirect (and correct the work if they redirect), and ask each "deferred-blocked" question and run its task once answered. These pending tasks come before step 8's queued work. If the queue is empty (the common case, `Status: empty`), note that and proceed. See the `## Attended-autonomous operating mode` section in [`.claude/CLAUDE.md`](../CLAUDE.md).

8. **Continue from "Next actions"** in the handoff file, honouring all standing disciplines (PRIMORDIAL RULE: Quality > Speed > Cost; post-commit audit before push; 60-second PR-activity fallback timer; formal per-PR `/validate-pr` + `/retro` for every non-handoff PR; the PR close-out checklist) and the step-5 decisions recorded for this session.

**Session metrics (standing, every session).** At this resume, capture `date -u` as the session start, and as the session runs accumulate the `subagent_tokens` reported by each `Agent` tool result (by phase: validation sweeps vs PR-build research vs other). At the session-closing handoff PR, write one row to [`.working/session-metrics.md`](../../.working/session-metrics.md) per its measured-versus-not-instrumented discipline (measured subagent tokens + elapsed wall-clock + PR/subagent counts; orchestrator main-loop tokens are `not instrumented`, never a fabricated figure). The row batches into the handoff PR's diff; it never goes in `CHANGELOG.md`. See the close-out checklist in [`.claude/CLAUDE.md`](../CLAUDE.md) `## Session migration and PR close-out checklist`.
