Resume the previous session from the durable handoff record. This is the single command the maintainer sends to continue work in a fresh session.

Execute these steps in order:

0. **Concurrency lease check (before reading the handoff or touching anything).** This is
   the two-part interlock from the "Session-concurrency safety" design in
   [`.working/design-decisions.md`](../../.working/design-decisions.md); gate 63
   (`tools/lint-session-state.py`) guards the lease file's shape, but the interlock
   DECISION lives here because CI runs per-branch and cannot see across sessions.
   1. `git fetch origin`, then read [`.working/session-state.md`](../../.working/session-state.md)
      from `main` (the declared half).
   2. If its `Status` is not `released` AND its `Last-heartbeat-UTC` is within the
      **60-minute staleness window**: a session is likely live. **HOLD and surface**
      ("branch X looks active, last heartbeat T, N minutes ago; confirm it is closed
      before I proceed"). This is an advisory HOLD requiring explicit maintainer
      confirmation; do NOT proceed on a timeout (the graceful-degradation timer never
      auto-proceeds here, because proceeding is the potentially destructive path).
   3. Cross-check git (the external half, the crash net for a session that died without
      releasing): list `origin/claude/*` branches ahead of `main`; for any branch other
      than the one being resumed, read its newest commit time. A commit inside the
      60-minute window triggers the same HOLD + surface.
   4. If the lease is `released` (or its heartbeat is stale beyond the window) AND no
      recent unmerged sibling branch exists: safe. A not-`released` lease with a STALE
      heartbeat is surfaced as an abandoned-session takeover decision ("prior session
      appears abandoned, heartbeat N old, branch unmerged; take over? (recommended) /
      investigate / leave it") rather than auto-blocking. On proceeding, ACQUIRE the
      lease: write `Active-session: <this branch>`, `Status: active`, and a fresh
      `date -u +%Y-%m-%dT%H:%M:%SZ` heartbeat (the write lands on `main` with the first
      PR; the stop-hook push makes the branch-local copy visible to the git cross-check
      in the meantime). Refresh the heartbeat at each PR close-out, optionally set
      `Status: winding-down` while assembling the session-closing handoff PR (a live
      state to this interlock, treated like `active`; gate 63 validates all three
      values), and RELEASE (`Status: released`, `Active-session: none`) in the
      session-closing handoff PR.

1. **Read [`.working/session-handoff.md`](../../.working/session-handoff.md) in full**, including its **"Known environment behaviours"** section. It is the as-of-last-refresh snapshot of branch, versions, counts, last-merged PRs, trust-recovery state, the next-actions queue, open decisions awaiting the maintainer, the known environment behaviours, and the standing disciplines. The known-behaviours section matters: the stop-hook auto-commits and pushes uncommitted changes on turn-end (the working tree is auto-persisted, not held locally), so verify `git log` rather than assuming the tree is uncommitted.

2. **Read [`.claude/CLAUDE.md`](../CLAUDE.md)** (the PRIMORDIAL RULE and project disciplines), the most recent few entries of [`CHANGELOG.md`](../../CHANGELOG.md) to ground recent history, and **[`.working/third-party-issues.md`](../../.working/third-party-issues.md)** (the log of execution-environment / third-party-service issues) so a recurring environment flake (e.g. a commit-signing-server 503 reddening gate 36) is recognized as an artefact, not chased as a corpus regression.

3. **Verify state against live files, do not trust the snapshot blindly** (the snapshot drifts forward between refreshes): run `git rev-parse --is-shallow-repository` (if `true`, `git fetch --unshallow` before any history-aware audit); `git log --oneline -5` to confirm HEAD; `tools/run_all_audits.sh` to confirm the corpus is green; and check the library/pack/README versions in `README.md` and `dev-security/claude-rules/README.md` against the snapshot. **In the same step, run the staged-brief freshness check** `python3 tools/audit-brief-freshness.py` (advisory, cross-repo; exits 0 always): if it reports the scratch coverage index 15-plus PRs behind main, dead brief target paths, or dead coverage-row TODO anchors, queue a targeted coverage-refresh sync (refresh the affected rows and briefs, not the whole index) as an early task; if the scratch checkout is absent in this environment, the tool says so and the check is a no-op. **Confirm the mechanical close-vs-start diff**: the `run_all_audits.sh` result on live `main` should match the snapshot's **green-at-`<sha>`** line (same gate count, same sha or a descendant). A mismatch here is the one genuine deterministic close-vs-start signal (e.g. a shallow-clone history-gate artefact, or a merge that dropped something), as distinct from the non-deterministic `/validate` subagent layer; investigate it before proceeding. **In the same step, run the execution-environment probe** `python3 tools/detect-env.py` (TODO 3.18) and act on its profile: pick the PR mechanism (GitHub MCP if `mcp__github__*` tools are in this session's tool list, the ASSISTANT-PROBE the script cannot see; else the `gh` CLI where the probe shows it authenticated), the CI-poll and merge transport (GraphQL when the probe shows headroom; the REST check-runs and merge endpoints when it reports the pool low, the #687 fallback), the commit/push mode (managed cloud sessions have the stop-hook auto-commit-push; local machines commit and push manually), and whether the pipe-guard hook fires (an ASSISTANT-PROBE: an unset `CLAUDE_PROJECT_DIR` blocks it on the cloud harness but NOT the NUC, so observe whether a piped verification is actually blocked; the unpiped-verification habit (RM-10) is the control either way). Verify sibling-repo access (`grc_library_ref`, `grc_library_scratch`); where access is missing, surface the probe's one-line `--add-dir` / `settings.local.json` fix to the maintainer, since `permissions` and `additionalDirectories` bind at session launch and are never self-granted mid-session.

4. **Surface a one-screen orientation to the maintainer**: current verified state (branch, HEAD, versions, gate/rule/skill counts), what merged most recently, and the next 1-3 planned actions from the queue. Note any drift between the snapshot and live state.

5. **Batch the unattended-run clarifications up front (maintainer-directed, 2026-06-25).** Before resuming the queue, ask the maintainer, via `AskUserQuestion`, batched into as few rounds as possible, every decision whose answer would let the session run unattended longer, so the assistant does not have to stop and ask mid-run. Pre-populate maintainer decisions for: (a) **all Priority 1 and Priority 2 backlog items** in [`TODO.md`](../../TODO.md) (the disposition, approach, and any ambiguous external value for each, drawing the candidate options from the item's recorded notes and a quick read of the target artefacts so the options are concrete); and (b) **the next 10-plus planned PRs**, including any that fall outside P1 and P2 (scheduled larger tracks such as the FR-167 batches, or P3-and-lower items the running order will reach). Group related items so the maintainer answers in a handful of structured questions rather than dozens; lead each with a recommended option. Record the answers in the next-actions queue (and, for durability across a mid-session compaction, in the working state) so they survive the rest of the session. Where the maintainer defers a decision, mark it deferred and route around it rather than blocking. The cost of one batch of clarifications at session start is far lower than repeatedly pausing an otherwise-unattended run. Apply the **compute-don't-ask** gate first: do not ask what the assistant can verify itself (versions, counts, file state); ask only the genuine authorial decisions.

6. **Run a full corpus-wide `/validate` as the first substantive task**, before resuming the queue. (It is fine to issue the step-5 clarification batch and then run this sweep while the answers come back; the sweep does not depend on the answers.) This is the compensating control for the session-closing handoff PR, which intentionally skips its trailing `/validate-pr` to avoid the post-merge validate-then-PR loop (a handoff PR's `/validate-pr` would produce ledger rows that need a new PR, whose merge would trigger another `/validate-pr`, with no terminating "next substantive PR" at a session boundary). The corpus-wide `/validate` is stronger than the skipped per-PR sweep would have been: it re-examines the whole corpus, not just the handoff PR's diff, so anything the prior session's closing PR(s) might have introduced is caught here. Record its findings per the `/validate` skill and route any to the backlog before continuing. **Cross-check the findings against the handoff's "Asserted expectations" section**: a finding that *contradicts* a surface the closing session asserted clean (a claimed-clean touched surface) is a genuine miss signal (the closing session's self-assessment was wrong), to be escalated rather than triaged as an ordinary out-of-window finding; a finding on a surface the closing session did not assert clean is an ordinary finding under the normal triage. This is why the closing session records asserted expectations rather than running a second `/validate` at close (the corpus is byte-identical across the boundary, so a two-run diff would fire on subagent sampling variance, not corpus facts).

6a. **Prune the handoff file in this first PR (the `/validate` close-out).** The session-closing handoff PR only ever *adds* per-session blocks, so [`.working/session-handoff.md`](../../.working/session-handoff.md) accumulates stale session history across resumes and buries the load-bearing state (its one job is a fast, reliable single resume point). As part of the first PR of the resumed session (the `/validate` close-out, which already touches `.working/`), prune the handoff per its `## Refresh and pruning discipline`: **keep current + 1 prior** in each per-session stack (the "Next actions" narrative blocks, the "State snapshot" blocks, the "Asserted expectations" blocks) and delete everything older, plus any superseded one-off "Next-session queue" / dated "This session's work" sections; **keep the standing sections in full** (header / `## How to resume` / `## Resume cursor`, `## Open decisions`, `## Trust-recovery state`, `## Known environment behaviours`, `## Standing disciplines`, `## Refresh and pruning discipline`). **Migrate-before-delete**: before deleting a block, confirm any load-bearing item it carries that is NOT recorded elsewhere (an open decision, a pending maintainer action) has been migrated to `## Open decisions` / [`TODO.md`](../../TODO.md) / [`pending-decisions.md`](../../.working/pending-decisions.md); pruned session *narrative* is durably recorded in `CHANGELOG.md` + its detailed mirror, `DONE.md`, `validate-sweeps/history.md`, `session-metrics.md`, and git history, so it is safe to drop. Maintainer-directed 2026-06-28.

7. **Resolve pending decisions and surface verifier overrides before the queue (attended-autonomous mode).** Read [`.working/pending-decisions.md`](../../.working/pending-decisions.md). If its queue is non-empty, resolve each entry BEFORE starting the next queued items: surface each "proceeded with stricter-safe default" entry to the maintainer for confirm-or-redirect (and correct the work if they redirect), and ask each "deferred-blocked" question and run its task once answered. These pending tasks come before step 8's queued work. If the queue is empty (the common case, `Status: empty`), note that and proceed. **In the same step, read [`.working/verifier-overrides.md`](../../.working/verifier-overrides.md)** and surface every `pending` override row to the maintainer for review (the finding, the overruling reasoning, and the recorded revert path), so an override made in a prior unattended run is cleared at the next attended boundary; a `pending` override is resolved to `reviewed` only by the maintainer, exactly as a pending decision is. If there are no `pending` overrides (the common case), note that and proceed. See the `## Attended-autonomous operating mode` section and the skeptical-verifier override discipline (in the [`ai-assistant-workflow-disciplines`](../rules/governance/ai-assistant-workflow-disciplines.md) pack rule) in [`.claude/CLAUDE.md`](../CLAUDE.md). In the same step, read [`.working/deep-assessment/register.md`](../../.working/deep-assessment/register.md) if it exists and surface any run row whose Status is `in-progress` (an unfinished `/deep-assessment` run resumes via `/deep-assessment` and is never silently dropped, exactly like the other standing registers).

8. **Continue from "Next actions"** in the handoff file, honouring all standing disciplines (PRIMORDIAL RULE: the AIQT Principle, (Accuracy = Integrity = Quality = Trust) > Speed > Cost; post-commit audit before push; 60-second PR-activity fallback timer; formal per-PR `/validate-pr` + `/retro` for every non-handoff PR; the PR close-out checklist) and the step-5 decisions recorded for this session.
