# Library Fitness Review, Run r1, 2026-06-21

## Executive Summary

This is the first invocation of `/fitness` against the GRC Documentation Library. Ten persona subagents were dispatched in parallel, each reviewing the corpus from an independent fresh-reader lens. Aggregate raw findings, 145; after dedupe and §3 enumeration with FR-N IDs, **111 unique findings**. Severity distribution by mechanical tabulation of §3: **17 high[critical] / 20 high / 57 medium / 17 low**.

*Note (corrected in Sweep 11 iter 1, 2026-06-21): the original PR #124 wording claimed "approximately 95 unique findings, 18 H[critical] / 22 H / 31 M / 24 L". That was an approximation during writing; the actual enumeration grew to 111 FR-N IDs as cross-cutting cluster findings were added during §3 authoring. The validation sweep caught the discrepancy. Counts above are now mechanically tabulated from §3 and are authoritative.*

**Bottom-line assessment**:

- **Fit for use, broadly yes**, with caveats below. Multiple personas independently confirmed the library's operational content is unusually strong: AI/agentic security is exemplar-grade, threat-modelling discipline is concrete and operationalised, OT/ICS coverage is comprehensive, post-quantum-cryptography roadmap aligns with NIST FIPS 203/204/205, supply-chain SCA/SBOM/SLSA guidance is current, breach response and DSAR workflows are production-quality, the three-lines-of-defence assurance map is structurally sound. The library exceeds the median quality of public GRC starter kits across most operational domains.

- **Fit for audit support, partial**. The procedural backbone is audit-grade (audit planning, control testing, CAPA, evidence packaging templates all present). Material gaps exist in (a) exception register lacks max-duration/renewal-count enforceability, (b) CAPA target-date extensions lack a governance ceiling, (c) compliance obligations register accepts low-precision regulatory citations, (d) per-control effectiveness metrics not defined (only programme-level metrics), (e) sub-control-level traceability for external-framework citations is incomplete.

- **Fit for executive consumption, with revision needed**. README opens with maintainer concerns (CalVer, version monotonicity) before stating the library's value proposition; dual-mission framing (GRC corpus + AI-assisted-maintenance reference implementation) confuses positioning; the "How to use" path sends executives to a 300-row register before the audience-keyed portal; one foundational role ownership reads as a category error (Enterprise Risk Management owned by CIO rather than CRO).

- **Fit for operational execution, yes for most procedures**. The "can I execute this end-to-end?" test is passed cleanly by supplier onboarding, change management, incident response, breach response, patch management, audit prep, DSAR, recovery, and tabletop exercises. Control testing is thinner than its peers and needs lifting; access control has decision-path gaps.

- **Fit for adoption, conditional on profile**. Mid-size SaaS targeting SOC 2/ISO 27001, EU financial services doing DORA, and logistics adopters operating CTPAT/AEO/BASC/PIP are well served. Healthcare HIPAA adopters, financial services outside EU/US (UK PRA/FCA, US OCC/FRB/FDIC, MAS, APRA, OSFI), global adopters with multi-jurisdiction AI Act obligations, and small organisations below E1 capacity face material friction.

- **Fit for newcomer onboarding, no**. The README never defines "GRC", offers no zero-knowledge entry path, sends new readers to a maintainer artefact (the document index), and the only worked example documents how to *write* corpus content rather than how to *apply* corpus content. The supporting newcomer materials exist (glossary, key-terms register, decision-tree dimensional questions) but discoverability is poor.

**Highest-risk issues** (the cluster a maintainer should prioritise this quarter):

1. **Maturity ladder fragmentation, three conflicting models**. `framework-governance-performance-and-improvement.md`, `template-maturity-self-assessment.md`, and `register-digital-trust-and-assurance-metrics.md` define materially different tier vocabularies. Adopter self-assessment is incoherent across surfaces.
2. **Data classification levels, 4-level vs 5-level split**. `standard-data-classification-and-handling.md` defines five (adds "Controlled"); `policy-privacy-and-data-governance.md` and `register-asset-inventory.md` reference four. Foundational concept inconsistency.
3. **DPO operational artefacts missing**. DPIA template/methodology, Article 28 Data Processing Agreement template, Privacy by Design operational artefact, Legitimate Interest Assessment template, Article 36 prior-consultation pathway, and Transfer Impact Assessment template are all referenced as required outputs but never provided.
4. **Audit-discipline ceilings absent**. Exception register has no max-duration; CAPA target-date extensions have no governance cap; risk-acceptance acceptance has no compensating-control field; compliance-obligations register accepts non-precise regulatory citations.
5. **Adopter-entry-point fragmentation**. README → document index, README → adopter guide, adopter guide → Tier-1 set (15 docs), quickstart → core baseline (6 docs), decision tree → 30/90/180 sequence, implementation roadmap → Phase 1/2/3. Six paths, no documented reconciliation.
6. **Retention contradiction**. `register-data-retention-schedule.md` sets SIEM event logs at 1y hot + 2y cold (3 years total). `standard-cloud-security-configuration-baseline.md` sets cloud activity log minimum at 90 days. Cloud logs feed SIEM; the 90-day floor at the source undercuts the 3-year baseline downstream.
7. **Foundational onboarding gap**. README never expands "GRC" despite using it in the title. No "if you are new to GRC, start here" block. The glossary defines it but is discoverable only by accident.

**Key remediation priorities**:

- **Quarter 1** (priorities surfacing audit / regulatory risk): close DPO operational template gaps (DPIA, DPA, LIA, TIA, PbD); reconcile data classification levels; reconcile maturity ladders; add exception/CAPA governance ceilings; fix SIEM/cloud-activity-log retention contradiction.
- **Quarter 1, parallel**: README rework, define GRC on page, add newcomer entry path, demote document index from primary navigation, fix dual-mission framing.
- **Quarter 2**: adopter entry-point reconciliation; healthcare HIPAA safeguard-level operational detail; FS-jurisdiction overlays (UK PRA/FCA, MAS, APRA, OSFI); AI jurisdiction annexes; editorial consistency sweep (must vs shall, role names, citation format); control-testing procedure lift.
- **Quarter 3**: domain extensions (crypto-asset/blockchain governance, M&A due diligence, sanctions/OFAC, AI ethics review process); upstream-sync maintenance documentation; sub-control traceability matrices.

**Publication-readiness**: not yet recommended. The DPO operational template gaps are visible to any privacy-savvy reader; addressing them is a quarter of focused work. Post-remediation the library is publication-grade.

**Personas dispatched**: 10. All ten returned findings. No persona was scope-skipped.

| # | Persona | Persona purpose | Findings returned |

| 1 | First-time executive reader | Trust, positioning, strategic comprehension | 10 |

| 3 | GRC practitioner | Risk/control ownership, framework alignment, maturity coherence | 13 |
| 4 | Auditor / assurance reviewer | Audit-readiness, testable controls, evidence paths | 15 |
| 5 | Policy and standards editor | Requirement language, naming, structural consistency | 16 |

| 7 | Skeptical reader | Ambiguity, contradictions, gaps, "in our org" leakage | 17 |

| 10 | Newcomer / onboarding engineer | Zero-knowledge entry-readability | 18 |

**Pages reviewed**: each persona strategically sampled the corpus per its lens. No persona claimed to have read every page. Persona coverage was complementary: security read security/ + dev-security/ + ai/ deep; DPO read privacy/ deep; GRC read governance/ + risk/ + compliance/ deep; adopter read docs/ + cross-domain navigation; auditor read governance/exception, compliance/, control-testing, evidence-packaging. Combined depth, approximately 70% of the corpus read in substance; remaining 30% spot-checked via directory listing or grep.

**Evaluation criteria applied**: each persona's brief specified persona-specific focus questions, evidence-requirement (quoted excerpt or path:line for every finding), explicit exclusions to prevent cross-persona overlap.

**Assumptions and limitations**:

- The library's external-standard citations were NOT independently verified against source documents (per `governance/specification-citation-verification.md` §14, library does not verify standard content vs interpretation). Findings claiming citation accuracy issues are based on the cited references being inconsistent across library documents, not on comparison to external sources.
- `.working/`, `.claude/`, `tools/`, `tests/`, CHANGELOG.md, TODO.md were excluded from all persona scopes (these are maintainer surfaces; not adopter-facing).
- The corpus contains approximately 308 markdown documents in domain directories plus 43 pack documents and 8 in docs/. Each persona prioritised high-impact pages within its lens; absent findings on un-sampled pages should not be interpreted as "in good shape" without spot-check.
- Persona findings carry persona-specific severity adjudication. Cross-persona dedupe sometimes raised severity (per Rule 5.3 pick-higher) when two personas surfaced the same defect through different lenses.

## Cross-Library Findings

Systemic patterns surfaced by multiple personas:

### Pattern 1: Editorial drift across same-document-type instances

The library's documents are well-formed individually but exhibit consistent drift across same-type instances. Three different H2 numbering conventions for policies; two different "Governance" section heading forms; "must" and "shall" used interchangeably with no documented register; NIST/ISO citation formats vary line-by-line. A unified style guide (currently absent) would close most of these by giving authors a single authority. Suggested artefact: `governance/specification-editorial-style.md` or a dedicated section in `specification-audit-programme.md`.

### Pattern 2: Operational template gaps in privacy domain

The privacy domain has 25 jurisdiction annexes (a real strength on breadth) but the corpus is missing roughly half a dozen operational templates that every privacy programme needs as core artefacts: DPIA template, Article 28 DPA template, LIA template, TIA template, Privacy by Design framework/standard, Article 26 joint-controller agreement template. Each is referenced as a required output but no template is provided. Adopters cannot operationalise GDPR Articles 25, 26, 28, 35, 36 from this library as it stands.

### Pattern 3: Maintainer mental model leaking into adopter-facing surfaces

Several surfaces address the maintainer (CalVer note in README §10, document index as "primary navigation", worked example documents ingestion not adoption, framework-document-architecture meta-content presented as domain framework) where they should address the adopter. The library's strongest content is operational; its weakest is meta-content presented as if it were domain-content. Separating maintainer-of-corpus mode from adopter-of-corpus mode in the docs/ tree would close this.

### Pattern 4: Audit-discipline ceilings absent

A consistent pattern across exception register (no max duration / renewal cap), CAPA target-date extensions (no governance ceiling), compliance obligations register (low citation precision accepted), risk register (no compensating-control field for acceptance), KPI/KRI escalation (no named decision-receiver). Each instance is an audit-finding magnet. A reviewer working through the corpus to confirm "what must be true / who is responsible / what evidence / how often reviewed" finds each control individually plausible but the governance ceiling missing.

### Pattern 5: Maturity-ladder fragmentation

Three different maturity ladders exist for the same notional concept (5-tier CMMI, 5-tier renamed, 4-tier with different vocabulary). Self-assessment depends on which ladder the adopter encounters. This is the most consequential cross-document inconsistency in the library.

### Pattern 6: Data-classification taxonomy fragmentation

The classification model has both 5-level and 4-level variants across foundational documents. Asset inventory, privacy policy, and security standard disagree on the canonical level set. Adopters labelling assets get inconsistent labels.

### Pattern 7: Adopter entry-point fragmentation

Six different "where to start" sequences exist. None document how they reconcile. An adopter selecting any one is implicitly choosing against the other five. The library's adopter-facing layer is structurally fragmented despite each individual entry document being well-shaped.

### Pattern 8: Coverage asymmetry across domains

Privacy domain: 25 jurisdiction annexes (breadth strong). Sector domain: 2 full operational annexes (DORA, SOX), 8+ named regimes with None/Referenced/Planned status. AI domain: 0 jurisdiction annexes despite 7+ domain documents. Healthcare domain: 1 annex covering all of HIPAA/HITECH/PHIPA/NHS-DSPT/etc. The library covers what it covers very well; the asymmetry is the pattern.

### Pattern 9: Tooling/role leakage without abstraction layer

"ITSM portal" / "ticketing system" / "SOC team" / "the CISO" / "the team" — used without an adopter-tooling-mapping register that documents which generic terms map to which adopter-side platform/role. The library wants to be tool-agnostic but does not institutionalise the mapping.

### Pattern 10: Newcomer-onboarding hole

GRC never expanded on the page that uses it in its title. No "if you are new to GRC, start here" path. Key newcomer-friendly artefacts exist (glossary, key-terms register, three-adoption-modes framing, decision-tree dimensional questions) but discoverability is poor. A 30-minute onboarding path would close this with low effort.

- Resolve the Classification/Confidentiality redundancy (Rec-16)
- Add `Adoption Disposition` field (Rec-9)

### Required page sections

Per-doctype section model needs documenting in the editorial style guide (Rec-15). Current variability (3 H2 numbering conventions for policies; Governance vs Governance-and-accountability heading) is the editor's biggest drift cluster.

### Requirement-language rules

Adopt RFC 2119:
- MUST / MUST NOT — absolute requirement
- SHOULD / SHOULD NOT — recommended; exceptions documented
- MAY — truly optional
- Deprecate "shall" in favour of "must"
- Deprecate "may not" in favour of "must not"

Closes FR-44, FR-45.

### Ownership model

The role authority register is the canonical source. Sweep the corpus to ensure all role mentions use the canonical full names (Chief X). Closes FR-46.

Define DPO as a distinct role in the register or document the convention that DPO is a sub-role of Chief Privacy Officer. Closes FR-47.

### Review cadence

Standardise on "Annual and upon material change" (the "and" form, not "or" form). Closes FR-52.

### Cross-linking model

Already reasonably enforced by gate 3 (link audit) and gate 11 (CHANGELOG link coverage). One gap: the `Related Documents` field is asymmetric by design (per the documented Decisions log); no change recommended.

### Versioning model

Per-document semver is well-enforced. Library-level CalVer is well-enforced. Consider whether `README Version` is a separately-useful concept or whether it should fold into library CalVer; current dual-versioning creates noise on the first screen of README (FR-8).

### Status labels

Not extensively used in the current corpus. If introduced, define values (Active / Draft / Deprecated / Retired) and the transition rules.

### Template recommendations

Privacy template suite needs the additions in Rec-3. Adoption guidance needs `worked-example-adoption.md` (Rec-10). Editorial style guide is itself a missing template (Rec-15).
