**Entanglement forms** (why each doc is deferred):
- **inbound `Â§N`** / **inbound "Section N"**: another live document cites this doc's section by number (a renumber shifts the target; the citer must be updated in the same PR). Both markdown-link (`[..](doc) Â§N`) and bare-parenthetical (`(doc Â§N)`) forms count. "Section N" citers include the [`dev-security/claude-rules/languages/`](../dev-security/claude-rules/languages/) pack files for the mobile-app standard.
- **intra `Â§N`** / **intra "Section N"**: the doc references its own sections by number in body prose (the reference must be remapped when the headings renumber).
- **inline clause numbering**: the doc's body carries numbered prose clauses (`3.1 Internal Audit must...`, `4.3 CAPA effectiveness must be validated...`) keyed to the OLD local H3 numbering; renumbering the headings without realigning these inline clauses (and any external citation of them, e.g. `procedure-capa.md` -> `policy-compliance-and-audit-management.md Â§4.3`) breaks the heading<->clause correspondence. The remap must renumber the inline clauses in lockstep with the headings.
- **uses "## Section N:" labels**: the doc's H2s are `## Section N: title`; the strip-and-renumber changes the section number the "Section N" scheme exposes, so it is unsafe wherever that doc is also cited or self-references.

**Build-time discipline.** Recompute the inbound citers at build time (they drift as the corpus changes); for each doc, produce the old-section-number to new-`Â§N` map and apply it to every citer (corpus-wide grep, not the doc's own input set) in the same PR, then re-parse. **Inbound-discovery recipe (#524 lesson):** discover inbound citers with a LINE-LEVEL grep over the full corpus (`grep -rnE '<basename>' --include='*.md' . | grep -E 'Â§|Section [0-9]|section [0-9]'`) plus a separate `<basename>.md#` anchor grep, never a proximity-bounded (character-window) pattern: a narrow window can return a false-negative when a `Â§N` sits far from the doc name in a table row (the #524 near-miss, where the first narrow scan missed `register-coverage-gaps Â§5`). Then distinguish an external-standard `Â§N` (e.g. `ISO/IEC 42001 Â§8`, which belongs to the cited standard, not the target doc) from a citation of the target doc's own section, and also run the exhaustive bare-numeric scan (`N.M`, "requirement/section/tier N") for docs whose numbers SHIFT rather than merely gain a prefix. **Inline-clause detection (#527 near-catch):** before classifying a doc as a safe prefix-add, scan the body for plain-text or bold numbered prose clauses (`grep -nE '^\*?\*?[0-9]+\.[0-9]'`) keyed to the section numbers (e.g. `1.1 Data Owners are responsible...` under `## 1.`); when present, the doc is the inline-clause hard class (the clauses must renumber in lockstep with the headings, and any external citation of a clause `Â§N.M` remapped), not a prefix-add. The near-catch: `security/standard-data-classification-and-handling.md` was carried as "intra Â§N" but a pre-renumber body scan found inline clauses `1.1`-`7.4` under a hybrid H2 scheme, so it was retagged inline-clause and deferred rather than renumbered. **Bare-external-Â§ disambiguation (#528 gate catch):** the intra-document-reference gate (`tools/lint-intra-doc-refs.py`) SKIPS a doc with no numbered headings, so a previously-unnumbered doc's OWN bare external `Â§N` cells (typically a framework-alignment table whose standard name lives only in the column header, e.g. `| Password requirements | Â§5.1.1 | ...` under a `NIST SP 800-63B` column) are never resolved before the renumber but collide with the new intra-document numbering the moment the doc is numbered, failing the gate. Before committing a renumber of a previously-unnumbered doc, grep the body for bare `Â§N` cells (`grep -nE 'Â§[0-9]'`) whose LINE does not itself name the external standard, and disambiguate each inline (`SP 800-63B Â§5.1.1`, matching the corpus convention e.g. `standard-logging-and-monitoring.md`'s `SP 800-92 Â§4.2` cell) in the same PR. This is a lockstep fix the renumber requires, not scope creep. `run_all_audits.sh` on the committed state is the backstop that catches it if the pre-commit grep misses it (as it did in #528, caught then fixed pre-push). Shape (maintainer directive 2026-07-01, supersedes the earlier "one domain-batch per PR" framing): **one document per PR** with maximal QA, no mass change until certain the renumber leaves zero broken links or references anywhere; each PR renumbers exactly one deferred document plus the citer files its inbound section-number references require. The numbering itself is gate-blind (gate 38 strips numbering when matching), so this is editorial consistency, not error-prevention. **H3-numbering convention (maintainer decision 2026-07-01, option B): number ALL H3s hierarchically (`### N.M`), including descriptive/unnumbered ones (e.g. `### Security and vulnerability testing` under `## 6.` becomes `### 6.1`), not only already-enumerated ones.** This resolves the earlier open A/B question (option A would have left descriptive H3s unnumbered); apply B to every remaining doc. The 5 docs merged before this decision already comply (their H3s were numbered or absent), so no retrofit is required. **H4 numbering (maintainer decision 2026-07-01): number ALL H4s hierarchically too (`#### N.M.K`), including descriptive ones** (e.g. `#### Minimum SBOM content` under `### 4.4` becomes `#### 4.4.1`), consistent with option B extended to the fourth level. **Section-N-label titles (maintainer decision 2026-07-01): when a doc uses `## Section N: lowercase title` H2 labels, convert to canonical `## M. Title` where M is the H2's DOCUMENT-ORDER position (not the old "Section N" number, which usually differs), and CAPITALIZE the stripped title to Sentence-case to match sibling canonical H2s** (e.g. `## Section 1: records retention schedule`, if it is the 5th H2 in document order, becomes `## 5.
