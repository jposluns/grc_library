## Subagent A return

Subagent A dispatched scoped to the #439 diff, the eight failure-mode classes plus a dedicated guard-logic review of the new shell script. Two findings:

### HIGH (guard-logic / class-d inferred-as-verified): `tools/pre-push-guard.sh:57-62` and `:66-71`

The original blocks used `if ! tools/run_all_audits.sh; then rc=$?; ... exit "${rc}"; fi`. `rc=$?` placed immediately after `if ! cmd; then` captures the exit status of the negated compound `! cmd` (which is `0` when `cmd` failed and the then-branch runs), NOT the runner's real non-zero status. Net effect: **the guard exited `0` when a runner failed**, so `tools/pre-push-guard.sh && git push` would proceed despite a failing gate. This defeated the guard's entire purpose and contradicted its own header ("Exit codes: 0 only if BOTH runners pass") and the #439 CHANGELOG's "refuses the push on non-zero" claim. The "self-tested green" claim in #439 exercised only the both-pass path; the failure path shipped unverified (a textbook evidence-grounded-completion failure: a state asserted without testing the path that would falsify it).

Empirically confirmed at apply-time: `f(){ return 7; }; if ! f; then rc=$?; echo $rc; fi` prints `0`; a faithful guard copy with a failing runner exited non-zero only after the fix.

### LOW (guard-logic): `tools/pre-push-guard.sh:54`

The best-effort fetch was hardcoded to `git fetch origin main`, so under a documented `BASE_REF=origin/develop` override it would refresh the wrong ref while the delta gates compared against the override base. Latent; harmless under the default base.
